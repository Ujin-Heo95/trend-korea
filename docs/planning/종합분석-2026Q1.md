## 1. 현황 종합 평가

| 서비스 기획 | **5/10** | 서비스기획 | "엔진은 좋은데 차체와 핸들이 부족" |

### 1.2 5개 부서 공통 합의

1. **v0.9.1은 이미 충분하다.** 기능 추가를 멈추고 사업 기반을 구축해야 한다.
2. **측정 없이 개선 없다.** Umami 분석도구가 모든 의사결정의 전제.
3. **도메인 없이 마케팅 없다.** SEO/AdSense/바이럴 전부 불가능.
4. **모니터링 없이 운영 없다.** Sentry/UptimeRobot 없이 프로덕션 에러 감지 불가.

### 1.3 달성한 것 (하루 만에 v0.1→v0.9.1)

- 소스 51개 활성 (커뮤니티 18 + 뉴스 14 + 테크 3 + 영상 1 + API 3 + 정부 2 + 기타)
- 3-Layer 중복제거 (MD5 + Jaccard + Thumbnail)
- 5-Factor 스코어링 (Z-Score + Velocity + Momentum + Trend Confirmation + Cluster Bonus)
- 교차 검증 (Google Trends x Naver DataLab x 커뮤니티 3중)
- 일일 리포트 MVP (Gemini Flash 요약)
- 영화/공연 탭 (KOBIS/KOPIS API, 포스터, 평점, 예매링크)
- 핫이슈 키워드 (Gemini Flash 추출)
- Discord 에러 알림, LRU 캐시, Railway 배포

### 1.4 아직 안 한 것

- 커스텀 도메인 없음 → SEO/AdSense/브랜드 전부 블로킹
- 개인사업자등록/개인정보처리방침 없음 → 수익화 불가
- 분석도구 없음 → DAU 측정 불가
- Sentry/UptimeRobot 없음 → 에러 추적 불가
- 공유 버튼 0개 → 바이럴 불가
- 프론트엔드 테스트 0%, E2E 0%
- PWA/이메일/카카오 채널 없음 → 리텐션 메커니즘 전무

## 2. 부서 간 교차 챌린지 — 핵심 쟁점 기록

### 쟁점 1: 기능 추가 vs 사업 기반 (CEO ↔ 개발 ↔ 마케팅)

| 부서 | 주장 |
|------|------|
| 개발팀 | "CI/CD + Sentry + Supabase에 2주 달라. 기술부채 70:신기능 30" |
| CEO | "기능은 충분하다. 도메인+Umami+Sentry 3가지만 하라" |
| 마케팅 | "공유 버튼 4시간이 스코어링 알고리즘 전체보다 사용자 획득에 임팩트 큼" |

**합의**: 생존 필수 기술부채(Sentry, DB 모니터링)만 즉시 처리. 나머지는 사업 기반과 병행.

### 쟁점 2: AdSense 승인 가능성 (사업 ↔ 서비스기획)

| 부서 | 주장 |
|------|------|
| 사업팀 | "어그리게이터 AdSense 승인 가능성 40-50%. 편집 콘텐츠 필수" |
| 서비스기획 | "일일 리포트 에디토리얼 격상 + /about 페이지 추가 필요" |
| CEO | "AdSense만 의존하지 말고 쿠팡 파트너스 즉시 병행" |

**합의**: Plan A(AdSense) + Plan B(제휴 마케팅) 동시 추진. 편집 콘텐츠 강화.

### 쟁점 3: 리텐션 구조적 결함 (서비스기획 ↔ 마케팅 ↔ 개발)

| 부서 | 주장 |
|------|------|
| 서비스기획 | "PostCard→외부이탈→재방문 동기 없음. 내부 이슈 상세 페이지 필요" |
| 마케팅 | "공유 버튼 + PWA + 카카오톡 채널이 먼저" |
| 개발팀 | "내부 상세 페이지는 2-3일짜리. Phase 3에서" |

**합의**: 1차 PWA+공유 버튼(즉시). 2차 내부 상세 페이지(Phase 3 후반).

### 쟁점 4: 카테고리 15개 과잉 (서비스기획 ↔ 개발)

| 부서 | 주장 |
|------|------|
| 서비스기획 | "15개→7-8개 통합. 빈약한 카테고리가 서비스 품질 인상을 깎음" |
| 개발팀 | "프론트엔드 탭 수만 줄이면 됨. 백엔드 변경 불필요" |

**합의**: 프론트엔드 탭 8개로 통합 (전체/커뮤니티/뉴스/테크/영상/핫딜/영화/공연).

### 쟁점 5: SEO에 SSR 필요 여부 (마케팅 ↔ 개발)

| 부서 | 주장 |
|------|------|
| 마케팅 | "네이버 크롤러가 JS 미실행. 네이버 검색 유입이 중요하면 SSR 논의 재개 필요" |
| 개발팀 | "Next.js 마이그레이션은 과잉. Fastify 동적 meta 주입으로 충분. 기각 유지" |

**합의**: 동적 meta 주입(3시간)으로 1차 대응. 네이버 검색 유입 데이터 확인 후 SSR 재검토.

## 5. 서비스기획 핵심 분석

### 5.1 사용자 여정 문제

```
유입 → TrendRadar(가로스크롤) → TrendingSection → CategoryTabs(15개!)
→ PostCard → 외부 사이트로 이탈 → 재방문 동기 없음 → END
```

핵심 결함: "출구만 있고 입구가 없는 건물" — 모든 콘텐츠가 외부 이탈 유도.

### 5.2 페르소나별 니즈

| 페르소나 | 핵심 니즈 | 현재 미충족 |
|----------|-----------|------------|
| 정보 중독자 (2030 직장인) | 한눈에 오늘 요약, 읽음 표시 | 일일 리포트 진입점 약함, 읽음 표시 없음 |
| 마케터/PR (3040) | 키워드 알림, 검색 저장, CSV 내보내기 | 알림 없음, 검색 결과 공유 불가 |
| 해외 한국인 | 일일 다이제스트, 시간대 필터 | 이메일/카카오 없음 |
| 개발자 | 다크 모드, 키보드 네비게이션, 테크 품질 | 다크 모드 없음, GeekNews disabled |

### 5.3 서비스기획 우선순위

1. **내부 이슈 상세 페이지**: 외부 이탈 방지 + 자체 콘텐츠 + SEO URL + 공유 랜딩
2. **카테고리 통합 (15→8)**: 정보 과잉 해소 + 서비스 정체성 명확화
3. **공유 버튼 + PWA**: 성장 엔진(바이럴) + 리텐션 트리거(홈화면)

## 부록: 각 부서 분석 원문

5개 에이전트가 독립적으로 작성한 분석 원문은 아래와 같은 구조로 수행됨:
- CEO: 전략 방향, 리소스 배분, 부서 간 우선순위 조율
- 서비스기획: 사용자 여정, UX 문제, 기능 우선순위, 페르소나
- 개발: 기술 건강도, 리스크, 스케일링 병목, 스프린트 계획
- 사업: 수익화 준비도, AdSense 리스크, 대안 모델, 법적/행정
- 마케팅: 준비도, 런칭 전략, 사용자 획득, 리텐션, 바이럴
